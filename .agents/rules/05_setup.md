# 環境構築・完了定義ガイドライン (Setup & Completion)

開発環境の運用ルールと、タスクを「完了」とするための最終チェック基準です。

## 1. 開発環境ポリシー

- **パッケージ管理**: `pnpm` を使用します（`npm` は使用禁止）。
- **並列開発**: `git worktree` を活用します。`pnpm wt:new <branch>` で作成し、`pnpm wt:dev` で起動してください。
- **環境変数**: `.env` はプロジェクトごとに独立して管理します。初回は `cp .env.example .env` で作成してください。

## 2. Prisma 運用ルール

- **マイグレーション**: 開発中は `pnpm prisma:migrate:dev --name <name>` を使用し、生成されたファイルをレビュー対象に含めてください。
- **適用**: `prisma db push` は原則禁止とし、`pnpm prisma:migrate`（deploy）による管理を徹底します。

## 3. タスク完了（提出）前のチェック

変更を提示する前に、以下のコマンドがすべてパスすることを確認（または実行を提案）してください。

1. `pnpm format` (整形)
2. `pnpm lint -- --fix` (静的解析と自動修正)
3. `pnpm typecheck` (型チェック)
4. `pnpm build` (ビルド確認)

## 4. 最終提出時のフォーマット

実装完了の報告時には、必ず以下の要素を含めてください。

- 実施した変更の要約（ビジネスロジック・UI等の変更点）
- 変更ファイル一覧
- 推奨される分割コミットメッセージ案
- チェック（Lint/Build）の実行結果
