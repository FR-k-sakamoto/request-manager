# Unit テスト生成

## AI協業フロー

```
1. 人間: テスト対象のモジュールやコンポーネントを指示
2. AI:   コードを読み、テスト要否を判断 → describe/it を日本語で生成し実装。純粋関数など入力パターンが多い場合は TEST_CASES 定数を分離したテーブル駆動で生成
3. 人間: 確認し、不足があれば describe/it を日本語で追記（中身は空でOK）、またはテーブル駆動の場合は TEST_CASES にケース（title/args/expected）を追記
4. AI:   追記された describe/it や TEST_CASES の中身を実装
5. 人間: テストの意図と実装の一致を確認
6. CI:   テスト実行・パス確認
```

## テスト対象の絞り込み基準

| 優先度   | 対象                   | 基準                                                 |
| -------- | ---------------------- | ---------------------------------------------------- |
| **必須** | 複雑なUsecase/Service  | 依存関数の呼び出しが3回以上、または条件分岐が3つ以上 |
| **必須** | 金額計算・日付処理     | ビジネスインパクトが大きい                           |
| **推奨** | バリデーションロジック | 入力パターンが多い                                   |
| **任意** | 単純なCRUD wrapper     | 基本不要。指示があれば作成                           |

## 社内コーディングルール

- **describe/it は日本語で記述する**
- **ファイル配置**: テスト対象と同じディレクトリに `*.spec.ts` として配置
- **1テスト1検証観点**: 1つの it に1つの検証観点。複数必要なら it を分ける
- AAA パターン（Arrange/Act/Assert）で記述
- テストデータはテスト内で完結させ、外部状態に依存しない
- 外部API（GeminiAPI等）は `vi.mock()` でモック化する。テスト内で実際のAPIを呼ばない
- **テーブル駆動テスト（TEST_CASES 分離）**: 同じロジックに対して複数の入力パターンを検証する場合、テストケースを `TEST_CASES` 定数として分離し `it.each` で実行する。仕様（データ）と検証ロジックを分離することで、ケースの追加・レビューが容易になる

  ```typescript
  // テストケース定義（仕様）
  const TEST_CASES = [
    { title: '整数の金額を通貨形式に変換できる', args: 1000, expected: '¥1,000' },
    { title: '0円を正しくフォーマットできる', args: 0, expected: '¥0' },
    { title: '小数点以下を四捨五入する', args: 1234.567, expected: '¥1,235' },
    { title: '負の金額にマイナス記号がつく', args: -500, expected: '-¥500' },
  ];

  // テスト実行（検証ロジック）
  it.each(TEST_CASES)('$title', ({ args, expected }) => {
    expect(formatCurrency(args)).toBe(expected);
  });
  ```

- テーブル駆動の使い分け:
  - **使う**: バリデーション、フォーマット変換、計算ロジック、条件分岐の網羅など、入力と期待値のペアが3つ以上あるケース
  - **使わない**: ケースごとにモック構成・セットアップが異なる場合や、`toThrow` / `toHaveProperty` など多様なマッチャーが必要な場合は個別の it で書く
